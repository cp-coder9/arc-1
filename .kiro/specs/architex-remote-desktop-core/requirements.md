# Requirements Document

## Introduction

The Architex Remote Desktop Core is a fully in-house remote desktop session layer that transforms confirmed marketplace bookings into live, governed remote access sessions. It builds on top of the existing Remote Desktop Marketplace (booking, catalogue, availability, reviews) by adding the missing execution layer: a Host Agent installed on resource owner workstations, a server-side Session Broker for signalling and policy enforcement, a browser-based WebRTC Viewer for renters, secure app isolation, controlled file handoff, and usage-based billing with full audit trail.

The Architex value proposition is the in-house broker, session policy engine, app-level isolation, and the integrated booking/payment/audit/FileManager workflow. Third-party infrastructure is used only where it accelerates delivery (TURN relay servers, GPU VMs for rendering) — no third-party VDI platforms (Guacamole, Parsec, etc.) are used for session delivery.

This module targets the South African market where bandwidth constraints require aggressive adaptation. It preserves all existing governance invariants: human approval required for bookings, auto-confirm prohibited, auto-payout prohibited.

## Glossary

- **Host_Agent**: A Windows-first desktop application installed on the Resource_Owner's workstation that registers the machine and its approved software with Architex, reports heartbeat, launches approved applications, and captures approved app windows for streaming.
- **Session_Broker**: The server-side component (Express 5 backend) that validates bookings, issues short-lived session tokens, orchestrates WebRTC signalling and TURN relay selection, enforces session policy, and writes audit events. Never exposes host credentials to renters.
- **Browser_Viewer**: The React-based in-browser component rendered within the Architex OS shell that receives the WebRTC media stream, displays approved application windows, forwards keyboard/mouse input, and shows session metadata (time remaining, file handoff status).
- **Resource_Owner**: A platform user (BEP, architect, firm) who installs the Host_Agent, publishes workstation resources, and approves bookings.
- **Resource_Consumer**: A platform user (freelancer, small practice, subcontractor) who books and connects to a shared workstation resource via the Browser_Viewer.
- **Session_Token**: A short-lived, single-use cryptographic token generated by the Session_Broker upon booking confirmation that authorises exactly one Resource_Consumer to connect to exactly one Host_Agent for the duration of the Booking_Window.
- **Booking_Window**: A confirmed time range during which a Resource_Consumer has permission to connect, as governed by the existing marketplace bookingService.
- **App_Allowlist**: A list of approved application executable paths configured per host that defines which programs the Host_Agent is permitted to launch and stream during sessions.
- **Session_Workspace**: A controlled directory on the host machine (e.g., `C:\ArchitexSessions\{sessionId}\`) where session output files are saved before handoff.
- **File_Handoff**: The process of transferring approved output files from the Session_Workspace into the Architex FileManager/project documents after session completion.
- **Grace_Period**: A configurable buffer (default 5 minutes, 0–15 minutes) after the Booking_Window end during which the Session_Broker warns the Resource_Consumer before enforcing disconnection.
- **Bandwidth_Adaptation**: Dynamic adjustment of stream encoding parameters (resolution, frame rate, bitrate, colour depth) based on measured network conditions between Host_Agent and Browser_Viewer.
- **Activity_Log**: A structured, append-only Firestore collection (`remote_desktop_session_events`) recording all auditable events during sessions.
- **TURN_Server**: A third-party or self-hosted relay server used when direct peer-to-peer WebRTC connections cannot be established due to NAT/firewall constraints.
- **Platform_Admin**: An Architex platform operator with oversight, dispute resolution, and session monitoring capabilities.

## Requirements

### Requirement 1: Host Agent Registration and Heartbeat

**User Story:** As a resource owner, I want to install an agent on my workstation that registers it with Architex and reports its availability, so that renters can see my machine is online and ready for sessions.

#### Acceptance Criteria

1. WHEN the Host_Agent is launched for the first time on a workstation, THE Host_Agent SHALL prompt the Resource_Owner to authenticate via Architex platform credentials and register the machine by writing a record to the `remote_desktop_hosts` collection containing: host ID, owner UID, machine name (maximum 64 characters), OS version, hardware specs (CPU model, RAM in megabytes, GPU model, storage in gigabytes), and registration timestamp.
2. WHILE the Host_Agent is running, THE Host_Agent SHALL send a heartbeat signal to the Session_Broker at intervals no longer than 30 seconds, containing: host ID, current status (idle, in_session, unavailable), CPU utilisation percentage, and available RAM in megabytes.
3. IF the Session_Broker does not receive a heartbeat from a registered Host_Agent for 90 seconds, THEN THE Session_Broker SHALL mark the host status as "offline" in the `remote_desktop_hosts` collection and update the associated marketplace listing session-readiness indicator to "unreachable".
4. WHEN the Host_Agent receives a heartbeat acknowledgement from the Session_Broker, THE Host_Agent SHALL synchronise the latest App_Allowlist and session policy configuration for the registered host, replacing any locally cached policy with the version indicated in the acknowledgement.
5. THE Host_Agent SHALL support Windows 10 (build 1903+) and Windows 11 as target operating systems for the initial release.
6. IF the Host_Agent detects that it is running without administrator privileges, THEN THE Host_Agent SHALL display a warning indicating that app isolation features require elevated permissions and continue operating with app isolation and sandboxing features disabled while all other functions (registration, heartbeat reporting, session brokering) remain active.
7. IF the Resource_Owner fails to authenticate during first-launch registration after 3 consecutive attempts, THEN THE Host_Agent SHALL display an error message indicating authentication failure and terminate without writing a record to the `remote_desktop_hosts` collection.
8. IF the Host_Agent fails to deliver a heartbeat to the Session_Broker for 3 consecutive intervals, THEN THE Host_Agent SHALL set its local status to "connection_lost", retry delivery on the next interval, and display a notification to the Resource_Owner indicating loss of connectivity to the platform.

### Requirement 2: Application Allowlist Management

**User Story:** As a resource owner, I want to specify which applications on my workstation are available for remote sessions, so that renters can only access the software I choose to share.

#### Acceptance Criteria

1. WHEN the Resource_Owner configures a host through the Host_Agent or the Architex web interface, THE system SHALL allow the Resource_Owner to specify between 1 and 20 application entries in the App_Allowlist, where each entry contains: display name (maximum 100 characters), executable path (maximum 260 characters), and software category selected from the platform-defined category list.
2. WHEN the Resource_Owner submits an App_Allowlist entry, THE system SHALL validate the entry by confirming the executable path references a valid `.exe` file on the host machine via Host_Agent local validation before accepting it into the `remote_desktop_apps` collection.
3. IF the Host_Agent validation determines that the executable path does not reference a valid `.exe` file on the host machine, THEN THE system SHALL reject the entry, retain the submitted form data, and display an error message indicating the executable path is invalid.
4. WHEN the Resource_Owner saves the App_Allowlist, THE system SHALL write each entry to the `remote_desktop_apps` collection with: app ID, host ID, display name, executable path, software category, and last-validated timestamp.
5. WHEN the Resource_Owner modifies the App_Allowlist for a host, THE system SHALL apply changes to future sessions only and SHALL NOT modify the allowlist of any session where a renter is currently connected.
6. IF a Host_Agent heartbeat reports that a previously validated executable path no longer exists on the host machine, THEN THE system SHALL mark that App_Allowlist entry as "unavailable" and notify the Resource_Owner via the Action Centre.
7. IF the Resource_Owner attempts to add an entry that would exceed the maximum of 20 entries in the App_Allowlist, THEN THE system SHALL reject the addition and display an error message indicating the maximum allowlist size has been reached.

### Requirement 3: Session Broker Token Issuance

**User Story:** As a platform operator, I want the session broker to issue short-lived tokens only for confirmed, governance-approved bookings, so that no session can start without proper authorisation.

#### Acceptance Criteria

1. WHEN a booking transitions to "confirmed" status in the marketplace bookingService (satisfying the `humanApprovalRequired` governance invariant), THE Session_Broker SHALL generate a Session_Token bound to the specific Booking_Window start and end times within 5 seconds of the status transition.
2. THE Session_Token SHALL contain: token ID, booking ID, consumer UID, host ID, Booking_Window start time, Booking_Window end time, Grace_Period duration, and a cryptographic signature that prevents tampering.
3. THE Session_Token SHALL expire automatically at the Booking_Window end time plus the configured Grace_Period, where the Grace_Period is between 1 and 30 minutes inclusive.
4. WHILE the Booking_Window has not started (current time is before Booking_Window start minus 15 minutes), THE Session_Broker SHALL reject connection attempts using the Session_Token with a "session_not_started" reason code.
5. IF a booking is cancelled after Session_Token generation, THEN THE Session_Broker SHALL revoke the associated Session_Token within 30 seconds and write a "token_revoked" event to the Activity_Log.
6. THE Session_Token SHALL be scoped to a single Resource_Consumer and a single host; the Session_Broker SHALL reject any attempt to reuse a Session_Token across different bookings, users, or hosts with a "token_scope_violation" reason code.
7. IF a Resource_Consumer requests a Session_Token for a booking whose governance status is not "confirmed", THEN THE Session_Broker SHALL reject the request and return a structured error indicating the denial reason (one of: "awaiting_owner_confirmation", "booking_conflict", "booking_cancelled", "booking_expired").
8. IF the Session_Broker receives a Session_Token whose cryptographic signature does not match the token payload, THEN THE Session_Broker SHALL reject the connection attempt with an "invalid_token" reason code and write a "token_integrity_failure" event to the Activity_Log.
9. IF the Session_Broker fails to generate a Session_Token due to a system error within the 5-second window, THEN THE Session_Broker SHALL return a structured error indicating "token_generation_failed" and the booking SHALL remain in "confirmed" status allowing retry.

### Requirement 4: WebRTC Signalling and Connection Establishment

**User Story:** As a resource consumer, I want to connect to the host workstation through my browser without installing software, so that I can start working immediately after my booking is confirmed.

#### Acceptance Criteria

1. WHEN a Resource_Consumer presents a valid, non-expired Session_Token to the Session_Broker, THE Session_Broker SHALL initiate WebRTC signalling between the Browser_Viewer and the target Host_Agent by exchanging SDP offer/answer and ICE candidates.
2. THE Session_Broker SHALL act as the signalling server only and SHALL NOT relay media streams; media flows directly between Host_Agent and Browser_Viewer via peer-to-peer or TURN relay.
3. IF a direct peer-to-peer connection cannot be established within 10 seconds, THEN THE Session_Broker SHALL provide TURN server credentials with a validity duration of no less than 5 minutes to both peers for relay-based connectivity.
4. THE Session_Broker SHALL never transmit host machine credentials (Windows login, IP address, RDP passwords) to the Resource_Consumer; the Browser_Viewer connects via the Session_Token and WebRTC only.
5. IF the WebRTC connection cannot be established within 30 seconds (including TURN fallback), THEN THE Session_Broker SHALL return a "connection_failed" error to the Browser_Viewer with a reason code (one of: "host_unreachable" when the Host_Agent does not respond to signalling initiation, "turn_unavailable" when TURN relay cannot be provisioned, "signalling_timeout" when SDP/ICE exchange does not complete) and write the failure event to the Activity_Log.
6. WHEN a WebRTC connection is successfully established, THE Session_Broker SHALL write a "session_started" event to the `remote_desktop_sessions` collection containing: session ID, booking ID, consumer UID, host ID, connection type (peer-to-peer or TURN relay), and start timestamp.
7. THE Session_Broker SHALL support a minimum of 50 concurrent active signalling sessions without signalling latency exceeding 2 seconds per SDP offer/answer round-trip relay.
8. IF the Session_Token expires during an in-progress signalling exchange before WebRTC connection is established, THEN THE Session_Broker SHALL abort the signalling process and return a "connection_failed" error with reason code "signalling_timeout" to the Browser_Viewer.

### Requirement 5: Host Agent App Capture and Streaming

**User Story:** As a resource owner, I want the agent to capture only the approved application windows and stream them to the renter, so that my desktop, personal files, and other programs remain private.

#### Acceptance Criteria

1. WHEN a session starts and the Session_Broker signals the Host_Agent with the session configuration, THE Host_Agent SHALL launch only the applications specified in the session's App_Allowlist (maximum 10 applications) and capture their window content for streaming, completing all application launches within 30 seconds.
2. WHILE a session is active, THE Host_Agent SHALL capture individual application windows (not the full desktop) using OS-level window capture APIs, transmitting only pixel data from approved application windows to the Browser_Viewer.
3. WHILE a session is active, THE Host_Agent SHALL hide the Windows taskbar, desktop icons, system tray, and Start menu from the captured stream, ensuring the Resource_Consumer sees only approved application windows.
4. IF the Resource_Consumer attempts to switch focus to a window not in the App_Allowlist, THEN THE Host_Agent SHALL block the input event, maintain focus on the last approved window, and write a "focus_violation_attempted" event to the Activity_Log.
5. WHILE a session is active, THE Host_Agent SHALL encode captured window content as a WebRTC video stream using H.264 or VP9 codec at 1080p resolution, 30fps, and adaptive bitrate between 1 Mbps minimum and 8 Mbps maximum (starting at 4 Mbps), with glass-to-glass latency not exceeding 150 milliseconds under stable network conditions.
6. IF a UAC prompt, system dialog, or administrator elevation request appears during an active session, THEN THE Host_Agent SHALL pause input forwarding to the session, hide the dialog from the stream, notify the Browser_Viewer that the session is temporarily paused with reason "system_dialog_detected", and terminate the session if the pause exceeds 60 seconds.
7. IF an application in the App_Allowlist fails to launch within 30 seconds or crashes during an active session, THEN THE Host_Agent SHALL write an "app_unavailable" event to the Activity_Log identifying the affected application, notify the Browser_Viewer of the application failure, and continue streaming the remaining approved application windows.
8. IF all approved application windows are closed or unavailable during an active session, THEN THE Host_Agent SHALL display a static placeholder frame to the Browser_Viewer indicating no active windows, suspend input forwarding, and write a "no_active_windows" event to the Activity_Log.

### Requirement 6: Browser Viewer Session Experience

**User Story:** As a resource consumer, I want a seamless in-browser remote desktop experience rendered within the Architex OS shell, so that I can work without leaving the platform or installing additional software.

#### Acceptance Criteria

1. WHEN a Resource_Consumer launches a confirmed booking from the marketplace, THE Browser_Viewer SHALL render an embedded session viewport within the Architex OS content area, receiving the WebRTC media stream from the Host_Agent.
2. WHILE a session is active, THE Browser_Viewer SHALL forward keyboard and mouse input events to the Host_Agent with latency no greater than the WebRTC round-trip time plus 50 ms processing overhead.
3. WHILE a session is active, THE Browser_Viewer SHALL display the remote application stream scaled to fit the available viewport while maintaining the original aspect ratio, adding letterbox bars when aspect ratios differ, with a minimum supported viewport size of 800×600 pixels.
4. WHILE a session is active, THE Browser_Viewer SHALL display a session control bar showing: elapsed time, remaining time within the Booking_Window (updated every 1 second), connection quality indicator (latency in ms, estimated bandwidth in Mbps), active application name, and file handoff status.
5. WHILE the Browser_Viewer has focus, THE Browser_Viewer SHALL intercept browser keyboard shortcuts (Ctrl+W, Ctrl+T, Ctrl+N, F5) and forward them to the Host_Agent; THE Browser_Viewer SHALL provide a dedicated escape key combination (Ctrl+Alt+Shift) to release focus back to the local browser.
6. WHEN the user attempts to navigate away, close the tab, or refresh the page while a session is active, THE Browser_Viewer SHALL display a confirmation dialog with the text "Leaving this page will disconnect your remote session."
7. WHILE a session is active, THE Browser_Viewer SHALL provide session controls: disconnect button, toggle fullscreen, quality profile selector (High: prioritize resolution up to source native; Balanced: cap resolution at 720p to maintain frame rate; Low: cap resolution at 480p to minimize bandwidth), and a "Report Issue" action that captures a timestamped event.
8. IF the WebRTC connection drops without a client-initiated disconnect, THEN THE Browser_Viewer SHALL display a reconnection overlay and attempt automatic reconnection every 5 seconds for up to 60 seconds (maximum 12 attempts).
9. IF automatic reconnection fails after 60 seconds, THEN THE Browser_Viewer SHALL mark the session as disconnected, display a message indicating the connection could not be re-established, and provide a "Return to Bookings" action.
10. WHEN the Booking_Window remaining time reaches zero during an active session, THE Browser_Viewer SHALL display a session-ending notification at least 60 seconds before expiry, and upon expiry SHALL gracefully disconnect the WebRTC stream and display a message indicating the booking period has ended with a "Return to Bookings" action.
11. IF the initial WebRTC connection cannot be established within 30 seconds of launching the booking, THEN THE Browser_Viewer SHALL display an error message indicating the remote host is unreachable and provide options to "Retry Connection" or "Return to Bookings."

### Requirement 7: Secure App Isolation and Input Control

**User Story:** As a resource owner, I want strict isolation so that renters cannot browse my file system, access other programs, or perform privileged operations during sessions, protecting my workstation security.

#### Acceptance Criteria

1. WHILE a session is active, THE Host_Agent SHALL block all keyboard shortcuts that could escape the approved application context, including: Alt+Tab, Win key, Ctrl+Esc, Alt+F4 (on windows not belonging to processes in the App_Allowlist), Ctrl+Alt+Del, and Task Manager shortcuts (Ctrl+Shift+Esc).
2. WHILE a session is active, THE Host_Agent SHALL prevent the Resource_Consumer from launching new processes, opening command-line shells (cmd.exe, powershell.exe, wt.exe, bash.exe, wsl.exe), or accessing Windows Explorer (explorer.exe) via any user-initiated input method including keyboard shortcuts, application menus, file dialogs, and drag-and-drop operations.
3. WHILE a session is active, THE Host_Agent SHALL disable clipboard transfer by default; WHERE the Resource_Owner has explicitly enabled text-only clipboard in the host configuration, THE Host_Agent SHALL permit text-only clipboard content (maximum 4096 characters) while blocking file, image, and rich-text clipboard content.
4. WHILE a session is active, THE Host_Agent SHALL restrict file dialog navigation (Open/Save dialogs within App_Allowlist processes) to paths within the Session_Workspace directory only, blocking traversal to any parent directory or path outside the Session_Workspace boundary.
5. IF the Host_Agent detects that a session process attempts to spawn a child process not in the App_Allowlist, THEN THE Host_Agent SHALL terminate the child process within 2 seconds of detection and write a "child_process_blocked" event to the Activity_Log including the blocked process name and parent process identifier.
6. IF a UAC elevation prompt or Windows Security dialog is triggered during an active session, THEN THE Host_Agent SHALL terminate the session within 3 seconds of detection, write a "session_terminated_uac" event to the Activity_Log, and notify both the Resource_Owner and Resource_Consumer with an indication that the session ended due to a privilege escalation attempt.
7. WHEN a session starts, THE Host_Agent SHALL verify that the App_Allowlist contains at least one entry and that the Session_Workspace path exists and is accessible, before granting the Resource_Consumer any input control.

### Requirement 8: FileManager Handoff

**User Story:** As a resource consumer, I want output files from my session to be securely uploaded to the Architex FileManager, so that I can access my work without needing direct access to the host machine.

#### Acceptance Criteria

1. WHEN a session starts, THE Host_Agent SHALL create a Session_Workspace directory on the host machine at a path configured by the Resource_Owner (default: `C:\ArchitexSessions\{sessionId}\`) and configure approved applications to use it as the default save location.
2. WHILE a session is active, THE Host_Agent SHALL monitor the Session_Workspace for new or modified files and report a file manifest (file name, size in bytes, last modified timestamp, file extension) to the Session_Broker at intervals no longer than 10 seconds.
3. WHEN a session ends (voluntarily or via auto-disconnect), THE Host_Agent SHALL compile a final file manifest and write it to the `remote_desktop_file_manifests` collection containing: session ID, booking ID, consumer UID, list of files (name, size, extension, SHA-256 hash), and manifest timestamp.
4. WHEN the Resource_Owner approves the file handoff (via a dedicated "Approve File Transfer" action in the Architex web interface), THE system SHALL upload each approved file from the Session_Workspace to the Architex FileManager, associating them with the booking's project reference if one exists, and SHALL set the transfer status of each file to "uploading" during transfer and "completed" upon successful upload.
5. IF a file in the Session_Workspace exceeds 500 MB, THEN THE system SHALL reject that individual file from the handoff, notify the Resource_Consumer that the file exceeds the transfer limit, and proceed with remaining files.
6. IF an upload to the Architex FileManager fails after 3 retry attempts (with a maximum timeout of 60 seconds per attempt), THEN THE system SHALL mark the affected file's transfer status as "failed", notify the Resource_Consumer that the file could not be transferred, and leave remaining files unaffected.
7. WHILE the session completion timestamp is less than 72 hours old, THE Host_Agent SHALL retain all files in the Session_Workspace; IF 72 hours elapse after session completion without Resource_Owner approval, THEN THE Host_Agent SHALL delete the Session_Workspace contents and write a "workspace_expired" event to the Activity_Log.
8. WHEN the Resource_Owner initiates the "Approve File Transfer" action, THE system SHALL present the file manifest with individual file selection (all files selected by default), allowing the Resource_Owner to deselect specific files before confirming the handoff.
9. THE Resource_Consumer SHALL be able to view the file manifest (file names, sizes, and transfer status showing one of: "pending_approval", "uploading", "completed", or "failed") in the Browser_Viewer session summary and in the booking detail view after the session ends.

### Requirement 9: Time-Bounded Sessions and Auto-Disconnect

**User Story:** As a resource owner, I want sessions to terminate automatically when booking time expires, so that consumers cannot overstay their allocated window.

#### Acceptance Criteria

1. WHILE a session is active, WHEN the Booking_Window end time minus the configured Grace_Period is reached, THE Browser_Viewer SHALL display a non-blocking countdown warning overlay showing remaining time in minutes and seconds, updated every 1 second, that does not prevent the Resource_Consumer from continuing to interact with the session.
2. WHEN the Booking_Window end time plus the configured Grace_Period is reached, THE Session_Broker SHALL terminate the active session by signalling both the Host_Agent and Browser_Viewer to disconnect, invalidate the Session_Token, and write an "auto_disconnect" event to the Activity_Log with reason "booking_window_expired".
3. THE Resource_Owner SHALL be able to configure the Grace_Period per resource listing between 0 and 15 minutes in 1-minute increments (default 5 minutes).
4. IF a Resource_Consumer disconnects voluntarily before the Booking_Window end time, THEN THE Session_Broker SHALL keep the Session_Token valid for reconnection until the Booking_Window end time plus Grace_Period is reached, allowing a maximum of 5 reconnection attempts within that window.
5. IF a Resource_Consumer disconnects after the Booking_Window end time but before the Grace_Period expires, THEN THE Session_Broker SHALL allow reconnection using the existing Session_Token until the Booking_Window end time plus Grace_Period, without resetting the countdown timer and counting against the same maximum of 5 reconnection attempts.
6. WHEN Auto_Disconnect is triggered, THE Host_Agent SHALL close all session-launched applications, finalise the Session_Workspace file manifest, and return to idle status within 30 seconds; IF the Host_Agent fails to reach idle status within 30 seconds, THEN THE Session_Broker SHALL force-terminate the session, write a "forced_disconnect" event to the Activity_Log with reason "host_cleanup_timeout", and mark the resource as requiring administrative review.
7. IF the Host_Agent is unreachable when Auto_Disconnect is triggered, THEN THE Session_Broker SHALL invalidate the Session_Token, disconnect the Browser_Viewer, write an "auto_disconnect" event to the Activity_Log with reason "host_unreachable", and queue a resource health check to be executed within 60 seconds.

### Requirement 10: Bandwidth Adaptation for South African Networks

**User Story:** As a resource consumer on a limited South African internet connection, I want the session to adapt to my available bandwidth, so that I can work productively even on constrained networks.

#### Acceptance Criteria

1. WHEN a WebRTC session is established, THE Host_Agent SHALL measure round-trip latency and available bandwidth within 5 seconds and select an encoding profile: "High" (1080p, 30fps, 4Mbps+), "Balanced" (720p, 24fps, 1.5–4Mbps), or "Low" (480p, 15fps, below 1.5Mbps).
2. WHILE a session is active, THE Host_Agent SHALL re-evaluate network conditions at intervals no longer than 5 seconds and adjust the encoding profile if sustained bandwidth crosses a profile boundary for at least 5 consecutive seconds, completing the profile switch within 3 seconds of confirmed threshold crossing.
3. THE Resource_Consumer SHALL be able to manually override the automatic quality profile by selecting from the predefined profiles (High, Balanced, Low) via the Browser_Viewer quality selector.
4. IF the Resource_Consumer has manually selected a quality profile, THEN THE Host_Agent SHALL suspend automatic profile switching and maintain the manually selected profile until the Resource_Consumer re-enables automatic adaptation or the session ends.
5. IF detected bandwidth drops below 500 Kbps sustained for 10 seconds and automatic adaptation is active, THEN THE Host_Agent SHALL reduce to a "Critical" mode (360p, 10fps, maximum compression) and THE Browser_Viewer SHALL display a warning indicating severely degraded connection quality.
6. WHILE a session is active, THE Browser_Viewer SHALL display a connection quality indicator showing current latency in milliseconds (rounded to the nearest integer) and estimated bandwidth in Mbps (to one decimal place), updated at intervals no longer than 5 seconds.
7. IF the Host_Agent is in "Critical" mode and detected bandwidth remains at or above 1.0 Mbps for at least 15 consecutive seconds, THEN THE Host_Agent SHALL exit "Critical" mode and re-select the appropriate standard profile (High, Balanced, or Low) based on current measured bandwidth.
8. WHEN a WebRTC session is established and the initial bandwidth measurement has not yet completed, THE Host_Agent SHALL default to the "Balanced" encoding profile (720p, 24fps) until measurement results are available.

### Requirement 11: Session Audit Trail

**User Story:** As a platform admin, I want comprehensive audit logging for every remote session, so that disputes can be resolved with evidence and accountability is maintained.

#### Acceptance Criteria

1. THE Activity_Log SHALL record the following events for every session: session_started, session_ended, app_launched, app_closed, file_created, file_modified, focus_violation_attempted, child_process_blocked, clipboard_used, auto_disconnect_triggered, reconnection_attempted, quality_profile_changed, and session_terminated_uac. Each event record SHALL contain: event type, session ID, booking ID, actor UID, actor role, host ID, and UTC timestamp with millisecond precision.
2. WHEN a session ends, THE Session_Broker SHALL write a session summary to the `remote_desktop_sessions` collection containing: total connected duration in seconds, total disconnection gaps in seconds, applications used (list of app names, maximum 100 entries), files created (count and total size in bytes), disconnection reason (one of: user_initiated, booking_window_expired, uac_terminated, connection_lost, owner_revoked, system_error), and connection type used.
3. THE Activity_Log SHALL be queryable by Platform_Admin (all sessions), Resource_Owner (sessions on their own hosts only), and Resource_Consumer (their own sessions only), with filtering by date range (maximum span of 12 months per query), session ID, host ID, and event type, returning results in pages of no more than 200 records per response.
4. THE Activity_Log records SHALL be append-only; no user role (including Platform_Admin) SHALL be able to modify or delete Activity_Log records during the retention period of 12 months.
5. IF the Activity_Log fails to persist an event during an active session, THEN THE Session_Broker SHALL retry the write up to 3 times with exponential backoff (1s, 2s, 4s), and if all retries fail, SHALL buffer the event in persistent local storage on the Host_Agent (up to a maximum of 10,000 events) and flush buffered events within 60 seconds of connectivity being restored, without terminating the session.
6. THE Activity_Log SHALL integrate with the Architex platform audit trail by emitting WorkflowEvents for critical events (session_started, session_ended, focus_violation_attempted, session_terminated_uac) to the Action Centre within 60 seconds.
7. IF a session ends abnormally (connection_lost or system_error) and the Session_Broker is unable to write the session summary, THEN THE Host_Agent SHALL persist sufficient session metadata locally to reconstruct the summary, and THE Session_Broker SHALL generate the summary from the persisted metadata within 5 minutes of the Host_Agent next connecting to the platform.
8. IF the Host_Agent local event buffer reaches its 10,000-event capacity before connectivity is restored, THEN THE Host_Agent SHALL discard the oldest buffered events first (FIFO eviction) and SHALL record a buffer_overflow event indicating the count of discarded events once connectivity is restored.

### Requirement 12: Billing Integration with Session Data

**User Story:** As a resource owner, I want session duration to be captured automatically from actual connection time, so that billing is based on real usage rather than estimated booking duration.

#### Acceptance Criteria

1. WHEN a session ends (voluntarily or via Auto_Disconnect), THE Session_Broker SHALL calculate actual connected duration as total time connected minus disconnection gaps of 60 seconds or longer, rounded up to the nearest minute, and report it to the existing marketplace billing pipeline within 30 seconds of session termination.
2. IF the Session_Broker fails to report the usage data to the billing pipeline within 30 seconds, THEN THE Session_Broker SHALL retry up to 3 times at 10-second intervals and, if all retries fail, SHALL flag the session record as "billing-pending" and surface an action item to the Resource_Owner via the Action Centre.
3. WHEN a session ends, THE Browser_Viewer SHALL present the Resource_Owner with a usage summary showing: booked window duration (minutes), actual connected duration (minutes), applications used, and files produced, within 5 seconds of session termination, and SHALL keep the summary accessible from the Action Centre until the usage log is finalised.
4. THE Resource_Owner SHALL retain the ability to adjust billed duration to any value between 1 minute and the total Booking_Window duration before finalising the usage log, preserving the `humanApprovalRequired` governance gate.
5. IF the calculated actual connected duration is zero minutes (all connection periods shorter than 60 seconds), THEN THE system SHALL display the actual connected duration as 0 minutes and SHALL require the Resource_Owner to explicitly set a billed duration of at least 1 minute or cancel the billing record before finalisation.
6. IF the Resource_Owner does not finalise the usage log within 48 hours of session completion, THEN THE system SHALL send a single reminder notification via the Action Centre; the billing record SHALL NOT be auto-finalised (preserving `autoPayoutProhibited` invariant).
7. WHEN the Resource_Owner finalises the usage log, THE system SHALL write a usage record to the `remote_desktop_sessions` collection containing: session ID, booking ID, booked duration minutes, actual duration minutes, billed duration minutes, owner-approved flag, and finalisation timestamp.

### Requirement 13: Platform Integration

**User Story:** As a platform architect, I want the Remote Desktop Core to integrate with Project Passport, SpecForge, Action Centre, FileManager, and the Analytics Engine, so that remote access activity is traceable within the project lifecycle.

#### Acceptance Criteria

1. WHEN a session completes and the booking is associated with a project reference, THE system SHALL write a session ProjectRecord to the Project Passport containing: session ID, booking reference, consumer UID, connected duration in whole minutes, applications used, files produced, and disconnection reason.
2. WHEN any of the following Activity_Log events occur — connection_failed, focus_violation_attempted, session_terminated_uac, or auto_disconnect_triggered — THE system SHALL emit a WorkflowEvent to the Action Centre within 60 seconds, including session reference, event type, affected project ID (if applicable), and assigned roles (Resource_Owner and Platform_Admin).
3. WHILE a Resource_Owner has pending booking confirmation requests, THE system SHALL display each request as an actionable item in the Action Centre showing host name, requested time window, and consumer name with approve and reject actions available. WHILE a Resource_Consumer has an active session, THE system SHALL display the session as an informational item in the Action Centre showing remaining time in minutes and host name.
4. WHEN a file handoff is completed (files uploaded to FileManager), THE system SHALL associate the uploaded files with the booking's project reference in the FileManager document registry, tagged with session ID and upload timestamp.
5. THE system SHALL expose session data to the Analytics & Reporting Engine for the following KPIs: utilisation rate (connected hours divided by available hours per host per calendar month), revenue per host (billing total per host per calendar month), session reliability (successful connections divided by total connection attempts per calendar month), and average bandwidth utilisation (mean percentage of allocated bandwidth consumed per session).
6. IF the system fails to write a session record to the Project Passport or FileManager (network error, permission denied, or target not found), THEN THE system SHALL queue the write for retry up to 3 attempts at 30-second intervals, log the failure to the Activity_Log, and notify Platform_Admin via a WorkflowEvent if all retries are exhausted.
7. IF the system fails to emit a WorkflowEvent to the Action Centre (network error or service unavailable), THEN THE system SHALL queue the event for retry up to 3 attempts at 30-second intervals and log the failure to the Activity_Log.

### Requirement 14: Existing Governance Preservation

**User Story:** As a platform operator, I want all existing booking governance guarantees preserved and extended to session-level enforcement, so that the trust model remains intact.

#### Acceptance Criteria

1. THE Session_Broker SHALL generate Session_Tokens only for bookings where the status is "confirmed" (booking confirmed by the Resource_Owner via the existing marketplace bookingService `confirmBooking` function, satisfying both `humanApprovalRequired` and `autoConfirmProhibited` invariants).
2. IF usage logs for a completed session have not been explicitly approved by the Resource_Owner (via a dedicated "approve usage" action) within 14 calendar days of session completion, THEN THE system SHALL prevent the associated billing record from being finalised and SHALL flag the record for platform operator review, preserving the `autoPayoutProhibited` invariant.
3. WHEN a booking conflict is detected by the existing marketplace `hasBookingConflict` function, THE Session_Broker SHALL refuse to generate Session_Tokens for the conflicting booking and SHALL return an error indication specifying the conflicting booking reference.
4. THE system SHALL preserve and extend the existing marketplace booking lifecycle (pending_owner_confirmation → confirmed → active → completed) by adding session-specific status transitions: confirmed → session_active → session_completed, where session_active is entered upon successful Session_Token generation and session_completed is entered upon explicit session termination by either the Renter or the Resource_Owner.
5. THE system SHALL NOT auto-confirm bookings, auto-generate tokens without owner confirmation, or auto-finalise billing records under any circumstance, regardless of session outcome or duration.
6. IF a booking status changes to "cancelled" or "revoked" while a Session_Token is in the "active" state, THEN THE Session_Broker SHALL immediately invalidate the associated Session_Token within 5 seconds and SHALL transition the session status to "terminated_governance", preserving any usage logs recorded up to that point.

### Requirement 15: Data Collections

**User Story:** As a platform developer, I want well-defined Firestore collections for remote desktop data, so that session state, events, and file manifests are persisted reliably and queryable.

#### Acceptance Criteria

1. THE system SHALL persist registered host machines in the `remote_desktop_hosts` collection with fields: host ID, owner UID, machine name, OS version, hardware specs (CPU model string max 128 characters, RAM in MB as integer, GPU model string max 128 characters, storage in GB as integer), status (online, offline, in_session), last heartbeat timestamp (UTC, millisecond precision), registration timestamp (UTC, millisecond precision), and configuration (Grace_Period in whole seconds ranging from 0 to 3600, clipboard policy as one of enabled or disabled, Session_Workspace path as string max 512 characters).
2. THE system SHALL persist allowlisted applications in the `remote_desktop_apps` collection with fields: app ID, host ID (referencing an existing document in `remote_desktop_hosts`), display name (max 128 characters), executable path (max 512 characters), software category (max 64 characters), validation status (valid, unavailable), and last-validated timestamp (UTC, millisecond precision).
3. THE system SHALL persist session records in the `remote_desktop_sessions` collection with fields: session ID, booking ID, host ID (referencing an existing document in `remote_desktop_hosts`), consumer UID, owner UID, project reference (optional, max 128 characters), status (pending, active, completed, terminated, failed), connection type (max 64 characters), start timestamp (UTC, millisecond precision), end timestamp (UTC, millisecond precision), total connected seconds (integer, 0 to 86400), total disconnection gap seconds (integer, 0 to 86400), applications used (array of app ID strings, max 50 entries), files produced count (integer, 0 to 10000), disconnection reason (max 256 characters), billed duration minutes (integer, 0 to 1440), and owner-approved flag (boolean).
4. THE system SHALL persist audit events in the `remote_desktop_session_events` collection with fields: event ID, session ID (referencing an existing document in `remote_desktop_sessions`), booking ID, event type (max 64 characters), actor UID, actor role (max 64 characters), host ID, timestamp (UTC, millisecond precision), and metadata object (max 8 KB serialized size, structure variable per event type).
5. THE system SHALL persist file handoff records in the `remote_desktop_file_manifests` collection with fields: manifest ID, session ID (referencing an existing document in `remote_desktop_sessions`), booking ID, consumer UID, owner UID, files (array of max 200 entries, each containing: name max 256 characters, size bytes as integer 0 to 10737418240, extension max 16 characters, SHA-256 hash as 64-character hex string, transfer status as one of pending, transferring, completed, or failed), manifest timestamp (UTC, millisecond precision), owner approval status (pending, approved, rejected), approval timestamp (UTC, millisecond precision, nullable), and expiry timestamp (UTC, millisecond precision).
6. IF a write operation to any remote desktop collection is missing a required field, THEN THE system SHALL reject the write and return an error indicating which fields are absent.
7. THE system SHALL support querying `remote_desktop_sessions` by host ID, by consumer UID, by owner UID, and by status, each returning results ordered by start timestamp descending.
8. THE system SHALL support querying `remote_desktop_session_events` by session ID ordered by timestamp ascending, and by host ID filtered by a timestamp range.

### Requirement 16: Session Recording Policy

**User Story:** As a resource owner, I want the option to enable session recording, so that I have evidence of what happened during a session in case of disputes.

#### Acceptance Criteria

1. WHEN configuring a host or resource listing, THE Resource_Owner SHALL be able to enable or disable session recording, with recording defaulting to disabled.
2. WHEN a Resource_Consumer connects to a host with session recording enabled, THE Browser_Viewer SHALL display a "Recording Active" indicator and require the Resource_Consumer to accept a recording consent prompt before any session data is streamed.
3. IF the Resource_Consumer declines the recording consent or does not respond to the consent prompt within 60 seconds, THEN THE Browser_Viewer SHALL cancel the connection attempt and display a message indicating that the session cannot proceed without recording consent.
4. WHILE session recording is enabled and a session is active, THE Host_Agent SHALL capture the visual output stream of the remote session as a recording stored on Architex-controlled infrastructure (not on the host machine), with a maximum recording duration of 8 hours per session.
5. THE session recording SHALL be viewable and downloadable only by the Resource_Owner, the Resource_Consumer for that session, and Platform_Admin, and none of those parties SHALL be able to delete a recording before its retention period expires.
6. THE system SHALL retain session recordings for 90 days after session completion, after which the system SHALL permanently and irrecoverably delete the recording.
7. IF a dispute referencing a session recording is open at the 90-day retention expiry, THEN THE system SHALL retain the recording until 30 days after the dispute status is changed to resolved or closed by a Platform_Admin.

### Requirement 17: Owner Session Controls

**User Story:** As a resource owner, I want to monitor and control active sessions on my workstation, so that I can intervene if something goes wrong.

#### Acceptance Criteria

1. WHILE a session is active on the Resource_Owner's host, THE Host_Agent SHALL display a local notification tray icon showing: session status (active/paused), consumer display name (truncated to 64 characters with ellipsis if longer), elapsed time in HH:MM:SS format updated every 1 second, and a "Terminate Session" action.
2. WHEN the Resource_Owner activates the "Terminate Session" action from the Host_Agent tray, THE Host_Agent SHALL signal the Session_Broker to end the session within 2 seconds, close all session-launched applications, and write an "owner_revoked" disconnection event to the Activity_Log.
3. THE Resource_Owner SHALL be able to view all active sessions and sessions completed within the last 30 days (up to 200 entries) on their hosts via the Architex web interface (Remote Desktop module), showing: consumer name, start time, elapsed duration, applications in use, and connection quality expressed as one of: good (latency ≤ 50 ms), fair (latency 51–150 ms), or poor (latency > 150 ms).
4. WHEN the Resource_Owner terminates a session, THE Browser_Viewer SHALL display a message to the Resource_Consumer within 3 seconds indicating the session was ended by the resource owner and provide a "Return to Bookings" action.
5. IF the Host_Agent loses connectivity to the Session_Broker while a session is active, THEN THE Host_Agent SHALL continue the session for up to 120 seconds; if connectivity is not restored within 120 seconds, THE Host_Agent SHALL terminate the session locally and write a "broker_connectivity_lost" event to the local buffer for later flush.
6. IF the Host_Agent fails to deliver the termination signal to the Session_Broker within 5 seconds after the Resource_Owner activates "Terminate Session", THEN THE Host_Agent SHALL terminate the session locally, write a "broker_unreachable_on_revoke" event to the local buffer, and display an indication to the Resource_Owner that the session was ended locally due to a connectivity issue.
