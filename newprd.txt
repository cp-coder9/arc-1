Main Risk
The platform is growing quickly, but the OS spine is not yet formalised enough. If more product features are stacked on top before stabilising architecture, the codebase may become harder to test, govern, secure, and safely extend with agents.

Amy's Priority Order
1. Stabilise the test suite.
2. Split api-router.ts into domain routers.
3. Create a formal Finding / Issue / Recommendation model.
4. Add versioned compliance rulesets.
5. Add project events and project activity streams.
6. Add permissioned agent action logs.
7. Build reports as structured project artifacts, not just chat summaries.
8. Harden municipal tracker security and credential handling.
9. Add a safe Architex agent API / MCP-style tool layer.
10. Add Revit Bridge later as a controlled read-only audit channel first.
11. Add project-scoped knowledge ingestion and a standards librarian / feedback loop.

Phase 2: OS Spine Models
Create first-class typed objects for:
- Finding / Issue / Recommendation
- Versioned compliance rulesets
- Project event model
- Project activity stream
- Agent action logs
- Report artifacts
- Built-environment discipline model

Required Event Types
Initial event model should include at least:
- document.uploaded
- drawing.review_requested
- issue.created
- issue.verified
- user.feedback.corrected
- report.generated
- revit_bridge.audit.completed
- municipal.status.changed
- agent.action.requested
- agent.action.completed
- agent.action.denied

Agent Governance Expectations
Agents must not mutate project data directly. They should operate through permissioned APIs/tool endpoints. Every agent action must be logged with actor, agent, project, tool name, input summary, output summary, approval status, risk level, timestamps, trace ID, and error status where applicable.

Municipal Tracker Expectations
Municipal tracker work must be security-first:
- encrypted credentials
- explicit consent records
- deletion support
- access control
- audit logs
- key-management clarity
- no credential leakage in UI/API responses
- agent access must be denied or heavily scoped

Revit Bridge Expectation
Start read-only only. First bridge should ingest or audit:
- model metadata
- sheets/views metadata
- titleblocks
- rooms/spaces
- schedules
- dimensions
- warnings/audit results
No model mutation until the read-only audit channel is proven, governed, and approval-gated.

Reporting Expectation
Graphic reporting must become a first-class artifact pipeline. Reports should be versioned project records linked to findings, rules, evidence, source documents, generation trace, and approval/review state.

Quality Gates Before Any Push/Deploy
1. git status --short --branch reviewed.
2. npm run lint passes.
3. npm test -- --reporter=dot passes.
4. E2E/smoke tests run where affected.
5. API/CORS probes pass for test.architex.co.za and api.architex.co.za if deployment is touched.
6. Backend/dashboard workflow smoke checks pass for affected routes.
7. No push/deploy without GMT approval.

Amy Bottom Line
The vision is strong and the codebase is alive, but the next phase must turn Architex into a reliable built-environment operating-system platform. Build the backbone first: tests, routers, findings, rules, events, reports, agent logs, safe APIs, municipal security, and controlled Revit audit integration.
