# Architex Web Platform Scope
## AI-Powered Operating System for the Built Environment
**First-Issue Project Scope / Web Platform Framework**  
**Prepared for:** Web Designer / Web Developer / Product Team  
**Platform:** Architex  
**Purpose:** Define the full platform scope, user roles, workflows, modules, tools, integrations, UX principles, AI orchestration layer, payment/escrow model, and governance requirements for development.

---

## 1. Platform Vision

Architex is an AI-powered operating system for the built environment. It connects clients, Built Environment Professionals (BEPs), contractors, subcontractors, suppliers, freelancers, municipalities, and administrators in one coordinated platform.

The platform is not only a marketplace. It is intended to become the operational backbone of built environment projects from project brief to close-out.

The central commercial drawcard is the marketplace where clients can post projects and hire verified Built Environment Professionals. However, the long-term value of the platform lies in its ability to manage the full project lifecycle through role-specific workflows, AI-assisted tools, payment/escrow infrastructure, compliance support, project management, procurement, resource sharing, and governance.

The system must be powerful enough to support built environment professionals and contractors, while remaining simple and intuitive enough for layman clients.

The guiding principle is:

> **One powerful platform underneath, with simple, guided, role-specific experiences on top.**

---

## 2. Core Positioning

Architex should be positioned as:

> **A guided built environment project platform that helps clients, professionals, contractors and delivery teams move projects from brief to close-out using AI-assisted workflows, verified participants, payment governance, compliance tools and project coordination.**

The platform should not feel like a complex database of modules. It should feel like a guided assistant that knows the built environment process and only shows each user what is relevant to them at the time.

---

## 3. Primary Platform Outcomes

Architex must support the following outcomes:

1. Clients can describe a project in simple language.
2. Clients can access verified BEPs and contractors.
3. Clients can manually search for known professionals or contractors, not only rely on AI matching.
4. BEPs can refine simple client briefs into technical scopes.
5. BEPs can coordinate full design teams.
6. BEPs can access compliance, drawing, municipal and project tools.
7. BEPs can post controlled work packages to freelancers.
8. Freelancers can support BEPs without accessing client marketplace projects directly.
9. Contractors can manage construction delivery with programme, Gantt, staff, wages, plant, procurement, subcontractors, claims and close-out tools.
10. Subcontractors and suppliers can participate at package level.
11. AI can assist with repetitive, administrative, compliance, coordination, procurement and reporting tasks.
12. Payments can be made through the platform using a gateway and escrow service.
13. Architex can monetise through nominal subscription fees, transaction/escrow fees, premium AI outputs, resource bookings and supplier/procurement value.
14. Admin can govern the whole system, including users, tools, payments, disputes, AI and platform settings.

---

## 4. Key UX Principle

The platform must prioritise ease of use.

The system is multi-layered, but the interface should not expose all complexity to all users. Each role must see only what is useful to that role.

A client must not see BEP admin tools, CPD tools, drawing management controls, municipal upload controls, subcontractor package administration or technical compliance workflows unless action is required from them.

A BEP must see design, compliance, municipal, professional coordination, contract, invoice, AI drawing, SANS form, resource, CPD and freelancer tools.

A contractor must see construction project management, staff, wages, plant, procurement, subcontractors, claims, invoicing, escrow and close-out tools.

A freelancer must see assigned work, submissions, drawing pre-checks, resource bookings, invoices and payments.

A subcontractor must see package-specific work, orders, delivery, claims, close-out evidence and payment status.

Admin must have governed access to the whole system.

## 7. User Roles

### 7.1 Client

A client is a layperson, property owner, developer or business needing built environment services.

The client can:

- Register and log in.
- Edit profile.
- Create a guided project brief.
- Upload photos, old plans, municipal documents, title documents and supporting files.
- Receive AI-assisted explanations.
- Receive educational feedback.
- View estimated project route, likely professionals, approvals and costs.
- View AI-matched BEPs and contractors.
- Manually search for known BEPs and contractors.
- Compare proposals.
- Appoint verified BEPs.
- Sign contracts.
- Pay invoices via payment gateway.
- Fund escrow milestones.
- View simplified project dashboard.
- View municipal status window.
- View contractor progress.
- View invoices, claims and escrow status.
- Query items contextually through linked messaging buttons.
- Participate in dispute resolution.
- View progress reports and close-out records.

The client should not:

- Access BEP CPD tools.
- Upload municipal tracker evidence unless specifically requested as a client action.
- Access BEP professional admin tools.
- Access freelancer marketplace jobs.
- Access subcontractor package controls unless payment/approval requires it.
- Access internal compliance administration.

BEPs can:

- Register and edit profile.
- Verify credentials via professional body API or manual verification.
- Access client marketplace opportunities.
- Pitch for client jobs.
- Be manually searched and invited by clients.

- Create fee proposals.
- Create client-professional appointment contracts.

- Manage drawing registers.
- Use AI drawing checker.
- Use SANS / compliance form autofill.
- Use municipal tracker.
- Upload municipal evidence.
- Link municipal API tracking where available.

- Rent out own resources through the resource sharing centre.

### 7.3 Freelancer

A freelancer is a qualified or skilled support person who is not necessarily fully registered as a BEP.

Freelancers can:

## 9. Guided Client Brief Wizard

Clients may not know how to structure a full professional brief. The platform must therefore provide a layman-friendly guided brief wizard.

The client brief wizard should:

- Avoid technical jargon.
- Ask simple, guided questions.
- Provide examples.
- Allow the client to upload photos, existing plans, title documents, municipal letters, WhatsApp images and other evidence.
- Ask what the client is trying to achieve.
- Ask whether the client has existing plans.
- Ask whether the work exists already.
- Ask urgency.
- Ask budget comfort level.
- Ask whether the client needs plans, approvals, construction pricing or full delivery support.
- Let AI explain likely next steps.

Example client options:

- “I want to renovate or add to my house.”
- “I need plans approved.”
- “I am selling and need updated plans.”
- “I want a construction price.”
- “I am not sure yet.”

The system should then generate:

- Client-friendly project summary.
- Possible project route.
- Likely professional requirements.
- Likely approval requirements.
- Possible risk flags.
- Suggested next action.
- Recommendation to invite verified BEPs.

- Clients with BEPs.
- Clients with contractors.
- BEPs with freelancers.
- BEPs with consultants.

- Projects with appropriate professional disciplines.

- Availability.

- Ratings.

- Project value range.

Manual search is essential because the built environment often works through word of mouth and trusted referrals.

The platform must allow:

- Clients to search for known BEPs and contractors.

- Users to search by:
  - Name.
  - Company.
  - Registration number.
  - Discipline.
  - Trade.
  - Region.
  - Verification status.

Manual search must not bypass verification. A user can search and invite a known professional or company, but the platform must still confirm verification and role eligibility.

Manual search results should show:

- Name or firm.
- Role.
- Discipline/trade.
- Region.
- Verification status.
- Ratings/reviews.
- Availability.
- Invite button.

Actions include:

- Invite to quote.
- Invite to tender.
- Invite to project.
- Invite to package.
- Invite to task.

## 17. SANS / Architectural Compliance Form Autofill

The system must create digital SANS and architectural compliance forms that can be auto-populated.

Data sources:

- Client project brief.
- Technical brief.
- Project details.
- Property details.
- Subscriber profile data.
- BEP professional registration data.
- Client profile data.
- Contractor profile data where applicable.
- Uploaded drawings.
- AI drawing interpretation.

The AI must assist by determining from uploaded drawings which form sections are applicable.

The workflow:

1. Project brief is completed.
2. Subscriber profiles provide repeated form data.
3. Drawings are uploaded.
4. AI reads drawings, schedules and notes.
5. AI detects applicable SANS/compliance sections.
6. AI fills digital forms.
7. Missing/uncertain fields become tasks.
8. BEP reviews and confirms.
9. BEP digitally signs where required.
10. Form pack is saved and issued/submitted.

Forms may include:

- Architectural compliance forms.
- SANS compliance declarations.
- Municipal application forms.
- Energy/fenestration schedule summaries.
- Fire/occupancy information.
- Discipline-specific declarations.
- Other compliance checklists.

Professional responsibility remains with the BEP.
